## [CRITICAL] Unauthentifizierter eingehender Handoff startet unbeschränkten Runner (RCE-Klasse)
**Modul:** agentlink · **Stelle:** `/home/till/claudeneu/core/src/hydrahive/agentlink/handoff_receiver.py:46-73`

**Problem:** Jedes ueber den WS empfangene Event mit type='handoff_received' und gueltiger state_id wird in lifespan._on_event (api/lifespan.py:159) an handoff_receiver.handle() weitergereicht. handle() laedt den State via get_state(), waehlt einen Ziel-Agenten und startet sofort den vollen Runner (_run_and_reply -> runner.run, Z.112-134) mit der angreiferkontrollierten task.description als user_input. Es gibt KEINE Authentifizierung des Absenders: weder der WS-Subscribe (_ws_listener.py:71) noch post_state/get_state (client.py) senden Token/Header; in settings/_services.py existiert kein HH_AGENTLINK_TOKEN. Wer den AgentLink-Service (oder den geteilten Redis/WS-Bus) erreicht, kann beliebige States posten und damit auf der HH2-Instanz Sessions erzeugen, die LLM+Tools ausfuehren. Da der Fallback-Ziel-Agent der Admin-Master ist (_find_target_agent, Z.92-93) und dessen Default require_tool_confirm=False ist (_config_utils.py:45, SPEC.md:63 'Unrestricted'), laufen shell/file/git-Tools ohne menschliche Bestaetigung — der angreiferkontrollierte Task-Text steuert direkt einen unbeschraenkten Agenten. Effektiv Remote Code Execution ueber einen unauthentifizierten Transport-Kanal.

**Fix:** Eingehende Handoffs vor Runner-Start authentifizieren und autorisieren: (1) shared secret/Token auf allen AgentLink-Calls (Authorization-Header in client.py, Token beim WS-Subscribe) ueber neues HH_AGENTLINK_TOKEN-Setting erzwingen; bei fehlendem/falschem Token Handoff verwerfen. (2) Den Admin-Master-Fallback in _find_target_agent entfernen — unadressierte oder unbekannte to_agent-Handoffs ablehnen statt auf den unrestricted Master zu eskalieren. (3) Fuer agentlink-stammende Sessions require_tool_confirm bzw. einen restriktiven Exec-Mode erzwingen, unabhaengig vom Agent-Default.

**Verifikation:** Der Befund ist vollständig bestätigt, keine Mitigation im Code vorhanden.  Konkrete Befunde im Code:  1. _ws_listener.py:67-71 — `websockets.connect(ws_url)` ohne Auth-Header; Subscribe sendet nur `{"action":"subscribe","channel":"agent:<id>"}`, kein Token.  2. client.py:61-76 — `post_state`/`get_state` nutzen `httpx.AsyncClient` ohne `Authorization`-Header.  3. settings/_services.py (_AgentLinkMixin) — kein `agentlink_token`-Property, kein `HH_AGENTLINK_TOKEN` wird gelesen. Der Env-Var-Name taucht ausschließlich in der Shell-Denylist (tools/shell.py:95) auf — er wird vor Agenten versteckt, aber nirgends gesendet.  4. lifespan.py:144-159 — `_on_event` leitet jeden `handoff_received`-Event mit state_id und ohne `reply_to:`-Reason direkt an `handoff_receiver.handle()` weiter. Kein Sender-Check.  5. handoff_receiver.py:80-93 — `_find_target_agent` fällt auf den Admin-Master zurück wenn `to_agent_id` unbekannt ist.  6. _config_utils.py:45 — `require_tool_confirm` default = `False`.  Der gesamte Sicherheitsperimeter liegt beim AgentLink-Service selbst. Ist dieser ohne Auth erreichbar (LAN-Segment, kompromittierter Host, fehlende Firewall-Regel), kann ein Angreifer einen beliebigen State posten, einen WS-Push auslösen und damit auf HH2 eine Session erzeugen, die der Admin-Master-Agent ohne Bestätigung abarbeitet — inklusive shell_exec, file-write, git. Die LAN-only-Beschränkung reduziert die Angriffsfläche auf interne Angreifer, kompromittierte Netzwerkteilnehmer oder Discord/WhatsApp-Bridges die als Relay missbraucht werden, schließt das Risiko aber nicht aus. Da zudem keine Discord/WhatsApp-Isolation für AgentLink-Channels besteht, ist der Vektor real.

## [CRITICAL] Projekt-Webhook führt Flows ALLER Owner aus (Tenant-Isolation gebrochen)
**Modul:** butler · **Stelle:** `core/src/hydrahive/api/routes/butler.py:147-149`

**Problem:** Der Endpoint POST /api/butler/webhooks/project/{project_id} authentifiziert nur gegen das Projekt-Secret und ruft danach `for flow in bp.list_flows():` ohne owner-Argument auf. `bp.list_flows(owner=None)` liefert die Flows ALLER Benutzer (persistence.py:47-66, iteriert alle owner-Verzeichnisse). Jeder dieser Flows wird mit `await bex.dispatch(flow, event)` real ausgeführt (dry_run=False default), wobei `event.payload` der vom Angreifer kontrollierte `request_body` ist. Wer das Webhook-Secret EINES Projekts kennt, triggert damit fremde Flows fremder User – inkl. Side-Effect-Actions (http_post, agent_reply, send_email-Stub). Verschärft: Der `webhook_received`-Trigger matcht bei leerem `hook_id` JEDEN Webhook (webhook_received.py:11-13), und das Projekt-Scoping (`channel=project:<id>`) wird beim Flow-Matching nicht gegen den Flow-Owner geprüft. Das ist ein klarer Cross-Tenant-/Authorization-Bypass im Self-Hosted-Multi-User-Setup (Rollen admin/user existieren, api_keys.py:70).

**Fix:** Flows strikt auf den Projekt-Owner scopen: statt `bp.list_flows()` nur Flows laden, die zu diesem Projekt gehören, z.B. `bp.list_flows(owner=f"project:{project_id}")` (das `project:`-Owner-Schema existiert bereits in persistence.py:21). Zusätzlich pro Flow prüfen, dass scope=="project" und scope_id==project_id, bevor dispatch läuft.

**Verifikation:** Der Befund ist vollständig bestätigt. In butler.py:147 wird `bp.list_flows()` ohne owner-Argument aufgerufen. persistence.py:54-55 bestätigt: `owner=None` iteriert explizit alle Owner-Verzeichnisse. Das `project_webhook`-Endpoint ruft danach `bex.dispatch(flow, event)` für jeden gefundenen Flow auf — keine Owner-Filterung, kein Scope-Check. Erschwerend: der Endpoint prüft `flow.enabled` NICHT (dieser Check existiert nur in `dispatch_event()` in executor.py:125, welches hier nicht genutzt wird). Zudem ist die Secret-Auth nur bedingt aktiv: `if expected_secret:` (Zeile 135) — fehlt das Secret in der Projektkonfiguration, ist der Endpoint ohne jede Authentifizierung zugänglich. Das korrekte Muster mit owner-Isolation existiert bereits in `executor.dispatch_event(owner=...)` — wird hier aber nicht genutzt. Im LAN-Betrieb mit Discord/WhatsApp-Inbound ist das real ausnutzbar: ein Angreifer, der das Webhook-Secret EINES Projekts kennt (oder ein Projekt ohne Secret findet), triggert Flows aller User inklusive Side-Effect-Actions (http_post, agent_reply). Fix: `bp.list_flows(owner=project_owner)` statt `bp.list_flows()`, wobei `project_owner` aus der Projektkonfiguration zu entnehmen ist — analog zu `dispatch_event(owner=...)`.

## [HIGH] Path-Traversal in VM-Update: raw iso_filename wird gespeichert und ungeprüft als QEMU -drive file= verwendet (Arbitrary Host File Read als CD-ROM)
**Modul:** vms · **Stelle:** `core/src/hydrahive/api/routes/vms_lifecycle.py:134`

**Problem:** Beim PATCH /api/vms/{vm_id} wird der ISO-Dateiname falsch validiert: resolve_iso(req.iso_filename) sanitisiert die Eingabe NUR fuer die Existenzpruefung (safe_filename -> Path(name).name, ersetzt Sonderzeichen, haengt .iso an) und prueft ob die sanitisierte Datei in vms_isos_dir existiert. In die DB geschrieben wird danach aber der ROHE Wert req.iso_filename (Zeile 134: iso_kw['iso_filename'] = req.iso_filename), NICHT der von resolve_iso zurueckgegebene sanitisierte Name. build_qemu_args (qemu_args.py:15) baut daraus dann ungeprueft iso_path = settings.vms_isos_dir / vm.iso_filename und mountet die Datei bei iso_path.exists() als '-drive file={iso_path},media=cdrom,readonly=on' in die VM.  Ausnutzung (durch jeden VM-Owner, kein Admin noetig): 1) Eigene ISO 'passwd.iso' in die Library hochladen. 2) PATCH /api/vms/{vm_id} mit iso_filename='../../../../etc/passwd'. resolve_iso sanitisiert intern zu basename 'passwd' + '.iso' = 'passwd.iso' -> Existenzpruefung besteht (die hochgeladene Datei). Gespeichert wird der Rohwert '../../../../etc/passwd'. 3) VM starten: vms_isos_dir / '../../../../etc/passwd' loest zu /etc/passwd auf, exists() ist true -> beliebige host-lesbare Datei wird read-only als CD-ROM in die Owner-VM gemountet und ist im Gast auslesbar. Damit umgeht ein nicht-privilegierter Owner die ISO-Library-Sandbox und liest beliebige fuer den Backend-Prozess lesbare Host-Dateien (z.B. /etc/passwd, Keys, andere VM-Disks). Auf dem create_vm-Pfad besteht der Bug NICHT (dort wird iso_safe = resolve_iso(...) gespeichert) - nur das Update ist betroffen.  QEMU-Comma-Injection ueber denselben Wert ist hier praktisch nicht ausnutzbar, weil sowohl resolve_iso als auch build_qemu_args (Zeile 44) eine reale Datei verlangen und ein komma-injizierter Pfad nicht existiert - die echte Schwachstelle ist die Path-Traversal-Datei-Exfiltration.

**Fix:** In vms_lifecycle.py den von resolve_iso zurueckgegebenen sanitisierten Namen speichern statt des Rohwerts. resolve_iso gibt bereits iso_safe zurueck:      elif req.iso_filename is not None:         iso_safe = resolve_iso(req.iso_filename)         if not iso_safe:             raise coded(status.HTTP_404_NOT_FOUND, "vm_iso_not_found", filename=req.iso_filename)         iso_kw["iso_filename"] = iso_safe  Damit landet ausschliesslich der basename-sanitisierte Name in der DB (analog zum create_vm-Pfad). Zusaetzlich defensiv in build_qemu_args (qemu_args.py:15) absichern: iso_filename erneut durch vmiso.safe_filename() schicken bzw. pruefen, dass iso_path.resolve() unter settings.vms_isos_dir.resolve() liegt, bevor die Datei als -drive gemountet wird (Defense-in-Depth gegen kuenftige Aufrufer).

**Verifikation:** Der Bug ist real und vollständig ausnutzbar. Der entscheidende Defekt ist in vms_lifecycle.py:131-134: resolve_iso(req.iso_filename) wird ausschliesslich wegen seiner Exception-Seite verwendet (wirft HTTP 404 wenn sanitisierter Name nicht existiert), der Rückgabewert (sanitisierter Name) wird verworfen. Stattdessen wird der rohe req.iso_filename in die DB geschrieben. Der Create-Pfad (Zeile 62-68) macht es korrekt mit iso_safe = resolve_iso(...). Die Exploit-Kette ist vollständig verifiziert: (1) safe_filename('../../../../etc/passwd') ergibt 'passwd.iso' via Path.name; (2) Existenzprüfung besteht wenn 'passwd.iso' in der Library liegt; (3) gespeichert wird '../../../../etc/passwd'; (4) build_qemu_args baut daraus settings.vms_isos_dir / '../../../../etc/passwd' = /etc/passwd; (5) iso_path.exists() ist True; (6) /etc/passwd wird als -drive file=...,media=cdrom,readonly=on an QEMU übergeben und ist im Gast per dd if=/dev/cdrom auslesbar. Auth (JWT-Bearer) und Owner-Check (vm_or_404) schützen nicht, da jeder authentifizierte VM-Owner seine eigene VM patchen darf. Inbound-Agenten und Discord/WhatsApp-Pfade machen das Angriffssurface erheblich breiter als ein rein manueller Zugriff. Der Fix ist trivial: Zeile 134 muss iso_kw["iso_filename"] = resolve_iso(req.iso_filename) lauten, analog zum Create-Pfad.

## [HIGH] Webhook-Secret-Vergleich nicht konstant-zeitig (Timing-Side-Channel auf das einzige Auth-Gate des Inbound-Endpoints)
**Modul:** communication · **Stelle:** `core/src/hydrahive/api/routes/communication_whatsapp_incoming.py:43`

**Problem:** Der einzige Schutz des untrusted Inbound-Endpoints `/api/communication/whatsapp/incoming` ist der Header-Vergleich `if x_hh_bridge_secret != expected`. Python `!=` auf `str` ist nicht konstant-zeitig — es bricht beim ersten abweichenden Byte ab. Es gibt keine globale Auth-Middleware (api/main.py: nur CORS, kein require_auth auf diesem Router) und KEIN Rate-Limiting (grep: kein slowapi/limiter im gesamten Core). Der Host ist über `HH_HOST` konfigurierbar (settings/_services.py:11-12, Default 127.0.0.1, in Prod typ. 0.0.0.0 / hinter Proxy). Wer den Backend-Port erreicht, kann das 43-Byte token_urlsafe-Secret per Timing-Orakel byteweise rekonstruieren. Mit dem Secret wird der Endpoint voll spoofbar: beliebige `external_user_id`/`participant` setzen und damit über wa_filter.evaluate `is_owner=True` erzwingen (Owner-Spoofing), beliebige `target_username` als Master-Agent ansprechen, oder den Agenten/STT mit Müll fluten (Kosten-DoS). Timing-Angriffe über LAN/Loopback sind praktisch nachweisbar; das fehlende Rate-Limit macht das Sammeln genügend vieler Samples trivial.

**Fix:** Vergleich auf `hmac.compare_digest` umstellen: `import hmac, secrets` und `if not expected or not hmac.compare_digest(x_hh_bridge_secret or "", expected): raise HTTPException(401, ...)`. Zusätzlich ein einfaches Rate-Limit / Fail-Counter pro Quell-IP auf diesem Endpoint, da er das einzige unauthentifizierte Loopback-/Inbound-Gate ist.

**Verifikation:** Der Befund ist technisch korrekt und real ausnutzbar, aber die Severity ist im Kontext zu reduzieren.  **Was der Code tatsächlich zeigt:**  1. `communication_whatsapp_incoming.py:43`: `if x_hh_bridge_secret != expected` — Python `str.__ne__` ist nicht konstant-zeitig. Bestätigt.  2. `api/main.py`: Kein globales Auth-Middleware auf diesem Router. Die Route ist unter `/api/communication/whatsapp/incoming` eingehängt ohne `Depends(require_auth)`. Bestätigt.  3. Kein Rate-Limiting: grep über den gesamten Core findet `slowapi`/`limiter` nur in unverwandten Dateien (llm, research). Bestätigt.  4. `ensure_secret` liest das Secret aus einer Datei — es ist ein `secrets.token_urlsafe(32)`, was 32 Bytes = ~43 Zeichen Base64url ergibt.  **Warum es ausnutzbar ist:**  Der `backend_internal_url` Default ist `http://127.0.0.1:{port}` (`_services.py:89`). Der WA-Bridge-Prozess läuft lokal und kommuniziert über Loopback. In der Standard-Installer-Konfiguration bindet der Core-Prozess auf `HH_HOST=127.0.0.1` (`installer/install.sh:33`). Der nginx-Proxy leitet `/api/` jedoch transparent durch (`60-nginx.sh:100-115`), auch den `/api/communication/whatsapp/incoming`-Endpoint, sodass der Endpoint über nginx von außen erreichbar ist (LAN/Internet je nach Firewall-Setup). Wer den nginx-Port (443/80) erreicht, kann den Endpoint ansprechen.  **Networking-Detail:** nginx enthält *keine* Blockierung für `/api/communication/whatsapp/incoming`. Der `/api/`-Block ist generisch und leitet alles weiter. Das bedeutet: der Endpoint ist über HTTPS von jedem LAN-Client erreichbar, der den nginx-Port erreicht.  **Praktische Ausnutzbarkeit eines Timing-Angriffs:**  Über LAN mit nginx-TLS-Termination als Proxy: Die zusätzliche RTT und TLS-Overhead durch nginx erhöhen das Jitter erheblich. Über Loopback wäre ein Timing-Angriff trivial (~nanosekunden Unterschied messbar), über LAN mit nginx ist es schwieriger aber nicht unmöglich — moderne Timing-Angriffe über das Netz sind mit genügend Samples dennoch nachgewiesen (vgl. Lucky13, Timing-Sidechannels auf HTTPS). Das fehlende Rate-Limiting ermöglicht das Sammeln der nötigen Samples (Tausende bis Zehntausende Requests) ohne Throttling.  **Was die Ausnutzung ermöglicht:**  Nach Secret-Rekonstruktion: Owner-Spoofing via `external_user_id`/`participant` → `is_owner=True` im `wa_filter.evaluate`-Call, beliebige Agenten-Ansprache via `target_username`, Kosten-DoS durch STT-Flooding.  **Mitigations die NICHT vorhanden sind:**  - `hmac.compare_digest()` statt `!=` — nicht vorhanden - Rate-Limiting — nicht vorhanden - IP-Whitelist in nginx für diesen Endpoint — nicht vorhanden - Der nginx-Proxy blockiert diesen Pfad nicht  **Severity-Anpassung:**  HIGH (nicht CRITICAL) ist angemessen. CRITICAL würde direkte RCE oder vollständigen Datenverlust bedeuten. Hier ist die Ausnutzbarkeit real, aber erfordert: (a) Netzwerkzugang, (b) statistisch aufwendigen Timing-Angriff über LAN, (c) das Secret ist 43 Bytes. Die Konsequenz (Owner-Spoofing, Kosten-DoS) ist ernst, aber kein vollständiger System-Compromise. Die Bewertung HIGH bleibt korrekt.

## [HIGH] Node-Bridge HTTP-Server ohne jegliche Authentifizierung (lokaler Steuerkanal: senden/verbinden/trennen offen)
**Modul:** communication · **Stelle:** `core/src/hydrahive/communication/whatsapp/bridge/index.js:25`

**Problem:** Der Bridge-HTTP-Server (`http.createServer`) prüft KEIN Shared-Secret auf eingehenden Requests. `HH_WA_BRIDGE_SECRET` wird nur in push.js:4 für AUSGEHENDE Push-Requests an das Backend verwendet, aber niemals für die EINGEHENDE Validierung in index.js. Damit kann jeder Prozess (oder jeder Request, der 127.0.0.1:8767 erreicht — z.B. via SSRF aus einem anderen Service, einem kompromittierten Plugin/Agent-Tool mit shell_exec, oder einem Container im selben Netz-Namespace) `POST /send/<user>` aufrufen und im Namen des verbundenen WhatsApp-Accounts beliebige Nachrichten an beliebige Nummern versenden, oder `POST /disconnect/<user>` aufrufen — was über sock.js:disconnect → clearAuth den gesamten Auth-State löscht (DoS: Account muss per QR neu gekoppelt werden). Der Server bindet zwar auf 127.0.0.1 (index.js:6), aber HydraHive führt selbst untrusted Agent-Tools (shell/HTTP) auf demselben Host aus — die Loopback-Grenze ist hier kein ausreichender Trust-Boundary.

**Fix:** Eingehende Requests in index.js gegen `process.env.HH_WA_BRIDGE_SECRET` prüfen (z.B. Header `X-HH-Bridge-Secret`), konstant-zeitig via `crypto.timingSafeEqual` auf gleich-lange Buffer. Requests ohne/mit falschem Secret mit 401 ablehnen, bevor connect/send/disconnect ausgeführt wird. Das Backend (whatsapp/adapter.py) muss den Header dann bei /connect, /disconnect, /send, /status mitsenden.

**Verifikation:** The finding is fully confirmed. index.js contains zero inbound authentication — no secret validation, no header check, nothing. HH_WA_BRIDGE_SECRET is injected into the bridge process (process.py:98) and read by push.js:4, but it is used exclusively for outbound push-requests from bridge to backend. There is no corresponding inbound validation in index.js. The fetch_url tool does block 127.0.0.1 via its SSRF guard (_BLOCKED_RANGES in fetch_url.py:31). However, the shell_exec tool (shell.py) provides unrestricted shell access with no localhost port blocking — curl/wget/nc can reach port 8767 trivially. An agent (or a prompt-injected instruction arriving via untrusted WhatsApp/Discord message) can call: `curl -s -X POST http://127.0.0.1:8767/send/till -H 'Content-Type: application/json' -d '{"to":"+49...","text":"..."}'` to send arbitrary WhatsApp messages impersonating the owner, or `curl -s -X POST http://127.0.0.1:8767/disconnect/till` to trigger sock.js:disconnect → clearAuth, deleting the entire auth state (DoS requiring QR re-pairing). Both attack paths require only that an agent executes shell_exec — a capability every agent in HH2 has by default. There is no mitigation in the codebase at the bridge level, at the adapter level, or at the shell_exec level that prevents this. Severity remains HIGH; not CRITICAL because exploitation requires either a compromised/malicious agent or a successful prompt injection through an untrusted incoming message, not direct external network access.

## [HIGH] System-Restore extrahiert mit filter="tar" statt filter="data" — beliebige Symlinks werden in Live-Systemverzeichnisse gepflanzt
**Modul:** backup · **Stelle:** `core/src/hydrahive/backup/restore.py:38`

**Problem:** `tar.extractall(extract_root, filter="tar")` verwendet den schwachen PEP-706-Filter. Empirisch verifiziert (Python 3.12.3): `filter="tar"` blockt zwar `..`-Pfadausbruch und Symlinks die AUSSERHALB des Zielordners LANDEN, erlaubt aber das Anlegen eines Symlinks INNERHALB des Zielordners, dessen ZIEL beliebig ist — sowohl absolut (`config/evil -> /etc/passwd`) als auch eskapierend-relativ (`sub/evil -> ../../../../etc/passwd`). `filter="data"` würde beides mit AbsoluteLinkError/LinkOutsideDestinationError ablehnen. Exploit-Kette: Angreifer-Archiv enthält Member `config/<name>` als Symlink auf z.B. `/etc/hydrahive/secrets` oder `../../tls/server.key`. Nach der sicheren Extraktion in das temp-Dir verschiebt `_atomic_replace_dir(cfg_extract, settings.config_dir)` (Zeile 48) das gesamte config-Verzeichnis inklusive des Symlinks per `shutil.move` 1:1 in das LIVE-Config-Verzeichnis — `shutil.move` dereferenziert den Symlink NICHT (verifiziert). Resultat: ein vom Angreifer kontrollierter Symlink liegt dauerhaft im Live-System und wird vom laufenden Dienst beim nächsten Config-Read gefolgt (Info-Leak fremder Dateien) bzw. beim nächsten Config-Write durchgeschrieben (Überschreiben beliebiger Dateien mit Dienst-Rechten). Gleiches gilt für die data/-Subdirs (agents/projects/plugins/whatsapp). Trigger: Admin importiert ein präpariertes "Migrations-/Backup"-Archiv (klassisches Restore-von-fremder-Quelle-Szenario).

**Fix:** In restore.py:38 `filter="tar"` durch `filter="data"` ersetzen — identisch zur bereits korrekten user_restore.py:42. Der data-Filter lehnt absolute/eskapierende Symlinks, Hardlinks, Device-Nodes und setuid/setgid-Bits ab und ist für Restore-aus-fremder-Quelle die einzig vertretbare Wahl.

**Verifikation:** Empirisch verifiziert (Python 3.12.3): `filter="tar"` lässt Symlinks mit absolutem Ziel (z.B. `/etc/passwd`) im Zielverzeichnis anlegen — kein Fehler, kein Block. `filter="data"` würde mit `AbsoluteLinkError` abbrechen. validate.py prüft ausschließlich Manifest-Version/Kind und SQLite-Integrität, kein Member-Type-Whitelist, kein Symlink-Scan. `_atomic_replace_dir` ruft `shutil.move` auf, das Symlinks nicht dereferenziert — der Symlink wird 1:1 ins Live-Config-Verzeichnis verschoben und bleibt permanent. Das einzige tatsächliche Gate ist `require_admin` auf dem `/api/admin/restore`-Endpoint (JWT, role=="admin"). Das mitigiert den Angriff durch externe Unbefugte, schließt aber das Hauptszenario nicht aus: ein Admin (oder ein Agent mit Admin-API-Key) importiert ein präpariertes "Migrations-/Backup"-Archiv — in HH2s Kontext mit autonomen Agenten und Inbound von Discord/WhatsApp keine rein hypothetische Situation. Nach erfolgreichem Restore liegt ein angreifer-kontrollierter Symlink dauerhaft im Live-System; beim nächsten Config-Read folgt der Dienst dem Symlink (Info-Leak), beim nächsten Config-Write überschreibt er die verlinkte Datei mit Dienst-Rechten (arbitrary write). Severity bleibt HIGH (nicht CRITICAL), weil der Exploit Admin-Authentifizierung voraussetzt und kein direkter Remote-Exploit ohne Auth-Bypass möglich ist.

## [HIGH] User-Restore überschreibt fremde Agents/Projects/Workspaces ohne Owner-Prüfung (Cross-Tenant-Tampering)
**Modul:** backup · **Stelle:** `core/src/hydrahive/backup/user_restore.py:45`

**Problem:** `restore_user_archive` ist über POST /api/users/me/restore für JEDEN authentifizierten User erreichbar (users.py:111-114, nur require_auth). Die einzige Eigentümer-Prüfung ist `manifest.username == username` (user_restore.py:37-38) — dieser Wert steht im vom Angreifer selbst erzeugten Archiv und ist trivial auf den eigenen Namen setzbar. Anschließend werden Agents/Projects/Workspaces NUR nach Verzeichnisname (= agent_id/project_id) eingespielt: `_restore_dir(tmp_path/"agents", settings.agents_dir)` (Zeile 45) und `_restore_projects` (Zeile 50, 114-130). Agents/Projects sind global per ID abgelegt (agents/_paths.py:42 `settings.agents_dir / agent_id`, projects/_paths.py:24), NICHT pro User namespaced. `_restore_dir` nutzt `shutil.copy2` (Zeile 111), das existierende Dateien überschreibt. Exploit: Angreifer (normaler User) baut ein User-Archiv mit `manifest.username = sein_eigener_name` und legt darin `agents/<fremde_agent_id>/system_prompt.md` bzw. `config.json` ab. Beim Restore wird der System-Prompt / die Konfiguration eines fremden Agenten überschrieben (Prompt-Injection, Tool-/Permission-Manipulation) bzw. fremde Projekt-Dateien injiziert. ## [MEDIUM] Container-Image-String wird unvalidiert als Argument an privilegierten `incus launch`-Subprocess gereicht (Argument-Injection-Härtungslücke)
**Modul:** containers · **Stelle:** `core/src/hydrahive/api/routes/containers_crud.py:52-54 (+ core/src/hydrahive/containers/incus_client.py:66)`

**Problem:** Beim Anlegen eines Containers wird `body.name` streng gegen `NAME_RE` (`^[a-zA-Z][a-zA-Z0-9-]{0,62}$`) geprüft (containers_crud.py:43), aber `body.image` NICHT. Pydantic erzwingt nur `min_length=1, max_length=120`. Der String durchläuft danach lediglich: `image = body.image.strip(); if ':' not in image: image = f'images:{image}'`. Anschließend landet er in incus_client.py:66 als `args = ['launch', image, name]` und wird via `asyncio.create_subprocess_exec('incus', *args)` an den incus-CLI übergeben — und zwar in einem bewusst privilegierten Kontext (security.privileged=true, security.nesting=true, incus_client.py:72; Backend läuft als root oder in incus-admin-Gruppe). Jeder authentifizierte Nicht-Admin-User darf laut SPEC (Per-User-Owner) Container anlegen, kontrolliert also `image` voll. Shell-Injection ist durch `create_subprocess_exec` (kein Shell) ausgeschlossen — `;`, `$()`, Newlines sind inert. Echtes Restrisiko ist Argument-/Flag-Injection in den incus-CLI: ein mit `-` beginnender Image-Token würde von cobra/pflag als Flag interpretiert. Der einzige Schutz ist hier der inzidentelle `images:`-Präfix (Tokens ohne Doppelpunkt werden präfixiert und damit zu harmlosen Positional-Args). Das ist KEINE bewusste Sicherheitskontrolle, sondern ein Nebeneffekt der Remote-Logik, und es gibt kein `--`-Separator vor den Positional-Args. Damit fehlt eine echte Allowlist-Validierung von User-Input direkt vor einem root-Subprocess. Kein sauber demonstrierbares RCE (Doppelpunkt-Barriere + pflag-Value-Syntax engen die Flag-Injection stark ein), aber eine klare Defense-in-Depth-Lücke an einer hochsensiblen Grenze.

**Fix:** Image analog zum Namen gegen eine explizite Allowlist/Regex validieren, BEVOR der `images:`-Präfix gesetzt wird, z.B. `IMAGE_RE = r'^(?:[a-z0-9-]+:)?[a-zA-Z0-9._/-]+$'` und in create_container nach dem strip() prüfen: `if not re.match(IMAGE_RE, image): raise coded(400, 'container_image_invalid')`. Zusätzlich in incus_client.launch ein `--` vor die Positional-Args setzen: `args = ['launch', '--', image, name]` (bzw. mit den `-c`-Optionen davor), damit incus die User-Werte garantiert nicht als Flags interpretiert.

**Verifikation:** Der Befund ist strukturell korrekt und nicht vollständig mitigiert. Die Analyse im Detail:  **Was wirklich schützt:** - `create_subprocess_exec` (kein shell=True) eliminiert Shell-Injection vollständig - Strings OHNE Doppelpunkt erhalten den `images:`-Präfix und landen als harmloser Positional-Arg — kein Flag-Parsing durch cobra/pflag möglich  **Was NICHT schützt:** - Strings MIT Doppelpunkt werden ungepräfixt weitergereicht. Ein Angreifer kann `body.image` auf z.B. `--target=remote-cluster:ubuntu/22.04` setzen (enthält `:`, kein Präfix, max_length=120 reicht locker). Cobra/pflag parsed `--target=...` dann als echtes Flag — positional vor den Positional-Args, kein `--`-Separator vorhanden. - Konkret ausnutzbare Flags: `--target` (Incus-Cluster-Redirection), `--project` (anderes Incus-Projekt), `--profile` (anderes Sicherheitsprofil). Diese können zwar keine direkte RCE auf den Host erzeugen, aber sie ermöglichen einem authentifizierten Non-Admin-User, Container mit abweichenden Profilen anzulegen oder auf Cluster-Targets umzuleiten — was über das intendierte Rechtesystem hinausgeht. - QUICK_IMAGES ist nur eine UI-Suggestion, keine serverseitige Einschränkung (Kommentar: "nicht ausschließend, User kann custom alias eingeben").  **Threat-Model-Einschätzung:** In einem reinen vertrauenswürdigen LAN mit ausschließlich manuellen Usern wäre das Restrisiko gering. Aber HH2s Threat-Model umfasst laut SPEC autonom schreibende Agenten und Inbound-Nachrichten von Discord/WhatsApp als unvertrauenswürdige Quelle. Ein kompromittierter oder manipulierter Agent, der einen Container-Create-Call absetzt, kann `image` frei wählen und damit Flag-Injection betreiben.  **Fix ist trivial und klar:** Eine Allowlist-Regex analog zu NAME_RE (z.B. `^[a-zA-Z0-9][a-zA-Z0-9./_-]*$`, max 120 Zeichen, kein `-` am Anfang) würde die Lücke schließen. Alternativ ein `--` vor den Positional-Args: `args = ["launch", "--", image, name]`.  **Severity-Anpassung:** MEDIUM bleibt korrekt. Kein direktes RCE, aber eine echte, ausnutzbare Defense-in-Depth-Lücke an einer root-Subprocess-Grenze mit nachgewiesenem Angriffsvektor (`:` im Image-String → Flag-Injection in incus CLI). Erhöhung auf HIGH wäre nur gerechtfertigt wenn ein konkreter Pfad zu Privilege-Escalation oder Cluster-Kompromittierung demonstrierbar wäre — das ist im vorliegenden Setup nicht eindeutig der Fall.

## [LOW] Restore-Endpunkte ohne Größen-/Dekompressionsbomben-Limit — Disk-Fill-DoS

Der Owner-Check in user_restore.py:35-38 verhindert das Wiedereinspielen fremder Daten, nicht aber eine crafted Bomb mit korrektem eigenen Manifest.
