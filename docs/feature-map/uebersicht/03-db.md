# Feature Map: Database — Datenbankschicht

> **Modul:** `core/src/hydrahive/db/`  
> **Was:** Gesamte Persistenzschicht. SQLite als Primär-DB, optionaler PostgreSQL-Mirror für Datamining.  
> **Warum:** Alle Sessions, Messages, Tool-Calls, Events, Health-Daten, FHIR — alles hier.

---

## Datenbankdateien

| DB | Pfad | Inhalt |
|---|---|---|
| SQLite Haupt-DB | `/var/lib/hydrahive2/db/hydrahive.db` | Sessions, Messages, Tool-Calls, Token-Stats, LLM-Calls, FHIR, Health, EGA, Errors |
| PostgreSQL Mirror | Optional (extern) | Datamining-Daten: Events, Embeddings, Graph |

## Datamining-Mirror (PostgreSQL)

Der Mirror ist eine separate Datenschicht die SQLite-Events in PostgreSQL repliziert
und zusätzliche Features wie Volltextsuche und Graph-Analyse bietet.

| Datei | Verantwortung |
|---|---|
| `mirror.py` | Mirror-Hauptmodul. Sync-Koordination. |
| `mirror_query.py` | Query-Layer für Mirror: Timeline, Suche, Semantic, Stats. |
| `_mirror_ddl.py` | PostgreSQL-DDL: CREATE TABLE Statements |
| `_mirror_writes.py` | Write-Operationen in Mirror |
| `_mirror_sessions.py` | Session-Sync-Logik |
| `_mirror_embed.py` | Embedding-Generierung für semantische Suche |
| `_mirror_search.py` | Volltext- + Vektor-Suche |
| `_mirror_explode.py` | JSON-Explode (Nested Content in suchbare Felder) |
| `_mirror_cards.py` | Card-Sync in Mirror |
| `_mirror_cards_model.py` | Card-Datenmodell |
| `mirror_graph.py` | Graph-Topologie: Knoten (Agents, Sessions) + Kanten (Handoffs, References) |
| `mirror_graph_topology.py` | Graph-Algorithmen (Connected Components, etc.) |
| `mirror_import_git.py` | Git-Commits in Mirror importieren |
| `mirror_import_jsonl.py` | JSONL-Dateien in Mirror importieren |
| `mirror_import_logs.py` | Log-Dateien in Mirror importieren |
| `mirror_import_shell.py` | Shell-Outputs in Mirror importieren |
| `mirror_import_sqlite.py` | SQLite-Daten in Mirror übertragen |

## Wichtige Tabellen (SQLite)

```sql
sessions         -- id, agent_id, user_id, title, created_at, first_kept_entry_id
messages         -- id, session_id, role, content (JSON), entry_id, created_at
tool_calls       -- id, message_id, session_id, name, args (JSON), result, success, duration_ms
llm_calls        -- id, session_id, model, input_tokens, output_tokens, cost_micros
compaction_events -- id, session_id, summary, first_kept_entry_id, created_at
token_stats      -- id, session_id, agent_id, user_id, input, output, cost_micros, date
fhir_resources   -- id, user_id, resource_type, resource_id, data (JSON)
health_metrics   -- id, user_id, metric_type, value, date
errors_log       -- id, session_id, agent_id, error, created_at
agent_handoffs   -- id, parent_session, target_agent, task, result, created_at
```

## Session-Lebenszyklus

```
sessions.create(agent_id, user_id)
  → session mit entry_id = 0, first_kept_entry_id = 0

messages.append(session_id, "user", content)
  → message mit entry_id++

messages.append(session_id, "assistant", content)
  → message mit entry_id++

# Nach Compaction:
sessions.update(session_id, first_kept_entry_id=N)
  → Nur Messages ab N werden an LLM gesendet
  → Alte Messages bleiben in DB (append-only!)
```
