# DB & Persistence

Subsystem-Pfade:
- `core/src/hydrahive/db/` — SQLite-Layer, Migrationen, CRUD-Module, PG-Datamining-Mirror, Importer
- `core/src/hydrahive/settings/` — Settings-Singleton (SSOT für Pfade/Env/Config), aus Mixins zusammengesetzt

Kernidee: **SQLite ist die autoritative Quelle** (Sessions/Messages/Tools + alle relationalen Domänen). Daneben gibt es einen **optionalen, fire-and-forget PostgreSQL-Mirror** (Datamining/Embeddings/Graph), der nur aktiv ist wenn `HH_PG_MIRROR_DSN` gesetzt ist. Der **Settings-Singleton** liefert jeden Pfad und jede Env-getriebene Konfiguration — nirgendwo werden Pfade hardcoded.

### DB-Kerninfrastruktur (`db/`)

- `init_db()` — Setup der SQLite-Datei (WAL, FK on, synchronous NORMAL) + alle Migrationen anwenden; idempotent, beim Start gerufen (`db/connection.py:14`)
- `db(immediate=False)` — Context-Manager: Per-Call-Connection mit FK-Enforcement, Auto-commit/rollback; `immediate=True` ⇒ `BEGIN IMMEDIATE` (Write-Lock vorab gegen TOCTOU) (`db/connection.py:31`)
- `apply_migrations(conn)` — Auto-Discovery: alle `migrations/*.sql` sortiert, `NNN_`-Prefix als Version, nur `version > current` werden ausgeführt (`db/migrations.py:29`)
- `current_version(conn)` / `_ensure_version_table(conn)` — `schema_version`-Tabelle (`db/migrations.py:14-26`)
- `uuid7()` — zeit-sortierbare UUIDv7 (RFC 9562) als String-PK (`db/_utils.py:9`)
- `now_iso()` — ISO-8601-UTC-Timestamp mit Millisekunden (`db/_utils.py:27`)
- Public-API-Re-Exports (`db/__init__.py`): `init_db`, `db`, `sessions`, `messages`, `tools`, `session_state`, `uuid7`, `now_iso`

## WIE

### Startup-Flow (DB-Initialisierung)
`api/lifespan.py:88-91`:
1. `settings.ensure_dirs()` — Verzeichnisse anlegen
2. `init_db()` (`db/connection.py:14`):
   - `settings.sessions_db.parent.mkdir(...)` (data_dir anlegen)
   - `sqlite3.connect(settings.sessions_db)`
   - `PRAGMA journal_mode = WAL`, `PRAGMA foreign_keys = ON`, `PRAGMA synchronous = NORMAL`
   - `apply_migrations(conn)` → Connection geschlossen
3. **nur wenn** `settings.pg_mirror_dsn` gesetzt: `await pg_mirror.init()` (Pool + PG-DDL + pgvector-Spalten)

### Migration-Algorithmus (`db/migrations.py:29-61`)
- `current = current_version(conn)` aus `MAX(version)` in `schema_version`
- `sorted(MIGRATIONS_DIR.glob("*.sql"))` — **lexikografische Sortierung der Dateinamen** ⇒ `NNN_`-Zero-Padding ist Pflicht
- Version = `int(f.name.split("_", 1)[0])`; ungültige Namen werden geloggt & übersprungen
- `version <= current` ⇒ skip
- `conn.executescript(sql)` — bei `OperationalError` mit `"duplicate column name"` oder `"already exists"` ⇒ Warnung + weiter (partiell angewendete Migration), sonst Re-Raise
- danach `INSERT INTO schema_version (version, applied_at)` + `commit()` pro Datei

### `db()`-Context-Manager-Semantik (`db/connection.py:31-50`)
- pro Aufruf neue `sqlite3.connect`, `row_factory = sqlite3.Row`, `PRAGMA foreign_keys = ON`
- `immediate=True` ⇒ `BEGIN IMMEDIATE` direkt nach Connect (Write-Lock vorab)
- `yield conn` → bei Erfolg `commit()`, bei Exception `rollback()` + Re-Raise, immer `close()`

### Read-Modify-Write mit `immediate` (`db/sessions.py:130-175`)
`set_model_override`/`set_reasoning_effort`: `db(immediate=True)` hält den Write-Lock über das gesamte SELECT-metadata → json-merge → UPDATE, danach `get()` + `mirror.schedule_session()`.

### Compaction-bewusste History (`db/_messages_llm.py:10`)
`list_for_llm()`: (1) letzte `compaction`-Message holen; (2) ohne Compaction → alle Messages; (3) `firstKeptEntryId` aus Metadata fehlt → alle non-compaction; (4) Timestamp des `firstKeptEntryId` nicht gefunden → alle non-compaction; (5) sonst `id >= first_kept` (UUIDv7-Stringsort = chronologisch) ⇒ behaltener Teil. Summary separat via `get_latest_summary()`.

### Message-Append + Mirror-Hook (`db/messages.py:13-52`)
`INSERT OR IGNORE` (Dedup über PK); nur bei tatsächlichem Insert (`rowcount > 0`): Session `updated_at` setzen + `mirror.schedule_message(m, s)`.

### Health-Ingest-Transaktion (`db/health.py:15-37`)
Roh-Insert in `health_ingest` UND Tages-Rollup `_process_payload_to_daily()` in **derselben** `db()`-Transaktion — schlägt der Rollup fehl, wird auch der Rohsatz zurückgerollt. Rollup-UPSERT: kumulative/zeitbasierte Metriken werden **summiert** (`value + excluded.value`), alle anderen **gemittelt** (`(value + excluded.value)/2.0`).

### PG-Mirror-Write-Pfad (fire-and-forget)
`sessions.create/update` + `messages.append` rufen sync `mirror.schedule_*()`. Diese versuchen `asyncio.get_running_loop().create_task(write_*)` — kein Loop ⇒ `RuntimeError` wird geschluckt (No-op außerhalb Async-Kontext). `write_message` (`_mirror_writes.py:34`): `explode()` → `executemany INSERT ... ON CONFLICT DO NOTHING` → `queue_embed()`.

### Embedding-Backfill-Schleife (`db/_mirror_embed.py:79`)
Iteriert non-embedded Events (`embedding IS NULL` + nicht-leerer Text), Batch `batch_size`, je `_EMBED_BATCH=32` ein API-Call (`aembed_batch`), `_store_batch` UPDATE nur `WHERE embedding IS NULL`. Abbruch wenn ein Batch 0 speichert (API-Fehler-Schutz) oder weniger Rows als batch_size. Wird per `on_embed_model_change` / Modellwechsel gestartet (`_start_backfill` loggt fehlenden Loop statt zu verschlucken).

### Embed-Spalten-Sync (`db/_mirror_ddl.py:207`)
`ensure_embed_col(conn, table)`: Dimension via `dim_for_model(embed_model)`; wenn vorhandene Spalte andere Dim hat → DROP+ADD `vector(dim)` + `embedding_model`/`embedded_at` + HNSW-Index (Fallback IVFFlat, Fallback Seq-Scan). `table` whitelisted auf `("events","cards")`.

### Graph-Berechnung (`db/mirror_graph.py`)
`build_graph()`: Rows mit Embeddings laden (`ORDER BY random() LIMIT MAX_NODES`), `<10` ⇒ leer; `_compute()`: numpy-Array → UMAP+HDBSCAN im Thread-Executor → Nodes (x/y/cluster) + `_cosine_edges` (normalisierte Dot-Products, Top-K über Threshold).

## WARUM (nicht-offensichtliche Verdrahtung, Invarianten, Fallen)

**SQLite ist autoritativ, PG-Mirror ist Beobachter.** Der Mirror ist reines Datamining/Embedding/Graph. Fällt `HH_PG_MIRROR_DSN` weg oder ist `asyncpg` nicht installiert (`mirror.py:36-40`), ist der gesamte Mirror ein No-op — Core läuft unverändert. **Keine Geschäftslogik darf je auf den Mirror angewiesen sein.**

**`NNN_`-Zero-Padding ist Pflicht.** `apply_migrations` sortiert die Dateinamen **lexikografisch** (`sorted(glob)`), nicht numerisch. `024` vor `100` funktioniert nur, weil dreistellig gepaddet wird. Eine `9_x.sql` würde nach `10_x.sql` einsortiert. Neue Migration = nächste freie dreistellige Nummer, monoton steigend. **Migrations-Dateien sind append-only — eine bereits ausgelieferte Migration nie editieren** (Version ist schon in `schema_version`, sie läuft nicht erneut).

**Migrations-Idempotenz hat zwei Mechanismen:** (1) `version <= current` skippt; (2) `duplicate column name`/`already exists`-Toleranz fängt partiell angewendete Migrationen (Script lief, aber `schema_version`-INSERT wurde nie committed). Deshalb verwenden viele neuere Migrationen `CREATE TABLE IF NOT EXISTS`/`CREATE INDEX IF NOT EXISTS` — sie müssen einen Re-Run ohne harten Fehler überstehen.

**UUIDv7 als PK ist load-bearing.** `list_for_llm` nutzt `id >= first_kept` statt `created_at >=`, weil UUIDv7 chronologisch als String sortiert und gleiche Millisekunden-Timestamps eindeutig auflöst. Würde man PKs auf zufällige UUIDs umstellen, bricht der Compaction-Cut. (Ausnahme: `llm_calls`/`compaction_events`/`errors_log` nutzen `secrets.token_hex`-IDs mit Prefix — die brauchen keine Sortierbarkeit, da sie über `created_at`-Spalte + Index sortiert werden.)

**`mirror.schedule_*` darf nie blockieren oder werfen.** Sync aufgerufen aus dem CRUD-Pfad, schluckt `RuntimeError` (kein Loop) und loggt sonst nur. Ein Mirror-Crash darf nie einen Session-/Message-Write scheitern lassen.

**`db()` öffnet pro Aufruf eine neue Connection** (kein Pool). WAL erlaubt parallele Reader + einen Writer. `BEGIN IMMEDIATE` (via `immediate=True`) ist nötig für Read-Modify-Write auf JSON-Metadata (Model-Override/Reasoning-Effort), sonst kann zwischen SELECT und UPDATE ein zweiter Writer reinschreiben (TOCTOU).

**FK ist nur enforced wenn `PRAGMA foreign_keys = ON` pro Connection gesetzt ist** — SQLite default ist OFF. `db()` und `init_db()` setzen es bei jedem Connect. Vergisst ein externer Codepfad das (z.B. eigener `sqlite3.connect` wie in `mirror_import_sqlite._read_*`), gelten CASCADE-Deletes nicht. Die Importer lesen nur, daher unkritisch.

**`compaction_message_id` ist bewusst KEIN FK** (`011_compaction_events.sql:40`) — ein FK würde beim CASCADE-Delete der Message das Event mitlöschen; die Telemetrie soll aber den Vorgang überleben.

**`tool_calls`-Telemetrie-Spalten (Migration 012) sind alle nullable ohne DEFAULT.** SQLite kann in `ALTER TABLE ADD COLUMN` weder FK noch non-NULL-DEFAULT auf bestehende Zeilen setzen. NULL bedeutet bewusst „wussten wir damals nicht" (≠ 0). Die `session_metrics`-VIEW filtert deshalb `WHERE session_id IS NOT NULL`, sonst zählten alte PR2-Zeilen falsch mit.

**`session_metrics` ist eine VIEW, kein Materialized.** Always-recompute beim SELECT — drift-frei, aber potenziell langsam bei vielen Sessions. Kommentar in `014_session_metrics.sql:3-4` sagt explizit: erst messen, dann optimieren (YAGNI).

**SQLite `session_metrics` ≠ PG `session_metrics`.** Der PG-View (`_mirror_ddl.py:161`) hat **keine** Tool-Aggregate, weil Tool-Telemetrie im Mirror in der `events`-Tabelle steckt (nicht in einer `tool_calls`-Tabelle). Die SQLite-Quelle ist authoritative für Tools. Wer Tool-Metriken braucht, muss SQLite abfragen.

**Naming-Drift SQLite↔PG:** SQLite-`sessions.user_id` wird im PG-Mirror zu `sessions.username` (`_mirror_writes.py:21-27`), und der PG-View mappt `s.username AS user_id` zurück. Events tragen `username` (= `s.user_id`). Wer die PG-Tabellen direkt abfragt, muss `username` statt `user_id` benutzen.

**Health-Rollup-UPSERT-Logik ist doppelt kodiert.** Die Liste der kumulativen/zeitbasierten Metriken steht sowohl als Python-Sets (`_CUMULATIVE`/`_TIME_BASED`, `health.py:78-84`) als auch hardcoded im `CASE WHEN metric_name IN (...)` des SQL-UPSERT (`health.py:138-143`). Wer eine Metrik hinzufügt, muss **beide** Stellen ändern, sonst summiert/mittelt es inkonsistent.

**Shell-Import filtert Secrets vor dem Schreiben** (`_SENSITIVE_RE`, `mirror_import_shell.py:26`) — Befehle mit password/token/key/bearer/credential werden nicht in den Mirror geschrieben. Dedup über `uuid5(NAMESPACE_URL, "shell:<user>:<day>:<cmd[:200]>")` ⇒ ein Befehl pro User pro Tag.

**Importer-IDs sind deterministisch (uuid5)** damit Re-Runs idempotent sind: git→`uuid5("git:<hash>")`, nginx→`uuid5("nginx:<ip>:<time>:<path>")`, journal→`uuid5("journal:<unit>:<line[:80]>")`, jsonl→`uuid5("jsonl:<session>:<id|ts>")`. Kombiniert mit `ON CONFLICT (id) DO NOTHING`.

**`secret_key` wirft beim ersten Zugriff** (`_services.py:19-23`), nicht beim Import. Da `@cached_property` lazy ist, crasht erst der erste JWT-Code-Pfad ohne `HH_SECRET_KEY` — nicht der Prozessstart.

**`@cached_property` friert Werte ein.** Settings werden beim ersten Zugriff aus der Env gelesen und gecacht. Ein `os.environ`-Change zur Laufzeit wirkt nicht mehr. **In Tests muss man Env vor dem ersten Settings-Zugriff setzen** — siehe Memory „Test-Gotcha: settings.data_dir Freeze": `hydrahive`-Imports lazy halten, sonst friert `settings.data_dir` zur Collection-Zeit ein und vergiftet die Session (`sessions_db` zeigt dann auf den falschen Pfad → 404/Setup-Fehler anderswo).

**`agent_name()` liest `agents/<id>/config.json` bei jedem Explode** (`_mirror_explode.py:89`) — synchroner File-Read im Mirror-Pfad, aber try/except-gekapselt mit Fallback auf die `agent_id`. Kein harter Fehler wenn die Config fehlt.

**Cards berühren nie das kuratierte Memory.** `wipe_cards`/`upsert_card` operieren ausschließlich auf der `cards`-Tabelle (abgeleitete, recompute-safe Schicht). `card_id = "card:<session_id>"` ⇒ genau eine Zeile pro Session, ON CONFLICT überschreibt. `computed_at` (now()) ist bewusst getrennt von `created_at` (Session-Zeit fürs Recency-Ranking).

**asyncpg liefert JSONB als Text-String** (kein Codec registriert). Cards-Read parst `topics/source/superseded_by/supersedes` manuell zurück (`_mirror_cards.py:40-49`), Write geht via `json.dumps` + `$::jsonb`.

**Patientenakte: Migration 023 ist DDL-Quelle, `patientenakte/schema.py` ist SSOT-Registry.** Die Registry mappt API-Felder auf Spalten + UI-Metadaten; Guard-Tests erzwingen, dass `ui_fields`-Keys als echte Spalten existieren (`PRAGMA table_info`). Wer eine `akte_*`-Spalte ändert, muss Migration UND Registry synchron halten (sonst „Agenten drehen Schleifen" — der Drift-Bug, den die SSOT-Umstellung gelöst hat).

## Offene Enden

- **Migrations-Kollisions-Risiko (#170):** Laut Memory soll #170 von `024_` auf `025_` umgenummert werden — aktuell ist `024_project_audit_log.sql` die höchste Migration; eine zweite `024_*` würde die lexikografische Reihenfolge brechen / mit dem belegten Slot kollidieren. Nächste freie Nummer = `025`.
- **Health-UPSERT-Doppelkodierung:** Die Metrik-Klassifikation steht doppelt (Python-Sets `_CUMULATIVE`/`_TIME_BASED` in `health.py:78-84` UND als hardcoded `IN (...)`-Liste im SQL `health.py:138-143`). Drift-anfällig — eine neue Metrik muss an beiden Stellen rein. Kandidat für eine geteilte Konstante.
- **`Card.confidence`/`superseded_by`/`supersedes` sind v2-Felder, in v1 ungenutzt** (`_mirror_cards_model.py:27,30`) — Schema bereits da, Logik fehlt. Bewusst geparkt (v2 Verify-Gate offen, siehe Memory „Proaktiver Recall").
- **`mirror_import_logs._parse_journal`** ruft `journalctl` ohne Privileg-Check — schlägt ohne Root/journal-Gruppe still fehl (returncode != 0 → leer). Kein Hinweis im Status-Dict, dass es an Rechten lag.
- **Naming-Drift SQLite `user_id` ↔ PG `username`** ist eine latente Falle für jeden, der den Mirror direkt abfragt (siehe WARUM). Nicht dokumentiert außer im Code-Kommentar.
- **PG-`session_metrics`-View fehlen Tool-Aggregate** (bewusst, aber inkonsistent zur SQLite-Version — wer cross-source vergleicht, bekommt unterschiedliche Spalten).
- **`mirror.recent_events` / `embed_status` / Importer-Status** sind alle modul-globaler Zustand (`_running`, `_progress`, `_pool`, `_backfill_running`) — nicht thread-/prozess-isoliert. Bei Multi-Worker-Deploy (mehrere Uvicorn-Worker) hätte jeder Worker eigenen Mirror-State + Pool; Status-Endpoints zeigten dann nur den State des antwortenden Workers.
- **`session_metrics` als always-recompute VIEW** ist explizit als „erst messen, dann optimieren" markiert (`014:3-4`) — skaliert linear mit Session-/Call-Zahl, potenzieller späterer Hotspot.
- **`agent_name()` liest `config.json` synchron pro Explode** (`_mirror_explode.py:89`) — File-I/O im Hot-Path des Mirror-Writes; bei vielen Messages viele redundante Reads (kein Cache).
- **`mirror_import_sqlite._read_*` öffnen eigene `sqlite3.connect` ohne `PRAGMA foreign_keys = ON`** — hier read-only, daher harmlos, aber ein Muster, das anderswo CASCADE-Verhalten unterlaufen würde.
- **`tool_calls`-PR2-Altzeilen mit `session_id IS NULL`** sind dauerhaft „verwaiste" Telemetrie ohne Session-Zuordnung — werden in `session_metrics` ausgefiltert, bleiben aber als Rauschen in der Tabelle.
- **`schema_version`-Tabelle hat kein `name`/`checksum`** — nur `version`+`applied_at`. Man kann nachträglich nicht erkennen, ob eine angewendete Migration inhaltlich der ausgelieferten Datei entspricht (kein Integritäts-Check gegen editierte Migrationen).
