Design — Health Feature

Vollständiger Plan zur Genehmigung. Passt alles?

1. Backend — neuer Endpunkt

Die bestehenden Endpoints liefern Rohdaten (JSON-Blobs). Wir brauchen einen neuen:

GET /api/health-data/metrics?days=7&metric=step_count

Parst alle gespeicherten Payloads on-the-fly, aggregiert Metriken pro Tag, liefert aktuellen Wert + 7-Tage-Trend. Buddy-Tool + Frontend nutzen denselben Endpunkt.

{
  "metrics": {
    "step_count": { "latest": 8432, "trend": "+12%", "unit": "count", "days": [...] },
    "heart_rate":  { "latest": 68,   "trend": "0%",   "unit": "bpm",   "days": [...] }
  },
  "last_ingest": "2026-05-15T22:49:00Z"
}

2. Buddy-Tool query_health_data

Neues Tool in core/src/hydrahive/tools/health_data.py. Buddy kann damit eigenständig Gesundheitsdaten abfragen und analysieren.

Parameter
metric — optional, Filter auf eine Metrik
days — Zeitraum (default 7)
Auth
Liest HH_HEALTH_API_KEY aus Settings — kein Key im LLM-Kontext

3. Frontend — features/health/

HealthPage.tsx — Hauptseite mit Split-View
_MetricCards.tsx — obere Karten (Schritte, Schlaf, HF, Kalorien + Trend)
_IngestList.tsx — untere Liste, Klick zeigt alle Metriken eines Pakets
_HealthBuddyBox.tsx — Box für BuddyPage (Status + 3 Buttons)
api.ts — API-Client für beide Endpunkte

4. Navigation + Routing

App.tsx
Route /health → HealthPage
Sidebar
Neuer Menüpunkt "Gesundheit" mit 🫀 Icon
BuddyPage
_HealthBuddyBox eingebunden, Buttons senden Prompt an Chat
Auth
Health-Seite nutzt bestehende Session-Auth — kein separater Key im Frontend
Scope: Kein Chart-Framework nötig — Trends als Text (↑ +12%) und Mini-Balken via CSS. Sauber, schnell, keine neue Dependency.
